## 3. Architecture discovery summary

The package implements a **two-graph LangGraph pipeline** with a Streamlit
UI on top, optional SQLite-backed tracing, and a deterministic review
layer that is currently a library — not yet a graph node.

### 3.1 Agents (`src/regrisk/agents/`)

All agents extend `BaseAgent` (`agents/base.py`) and accept an
`AgentContext` carrying `client`, `model`, `max_tokens`, `temperature`,
plus per-call config. The pattern is **LLM-first with a deterministic
fallback** so the pipeline runs without API keys.

| Agent | File | Purpose |
|---|---|---|
| `ObligationClassifierAgent` | `obligation_classifier.py` | Classify each obligation group into one of 5 categories × 5 relationship types × 3 criticality tiers. |
| `APQCMapperAgent` | `apqc_mapper.py` | Map each classified obligation to ≤ `max_apqc_mappings_per_obligation` APQC process nodes. |
| `CoverageAssessorAgent` | `coverage_assessor.py` | Per (obligation, candidate-control) pair, score structural + semantic + relationship match → `Covered` / `Partially Covered` / `Not Covered`. |
| `RiskExtractorAndScorerAgent` | `risk_extractor_scorer.py` | For each gap, extract 1–3 risks with impact × frequency scoring (4-point scales). |
| `ControlImprovementAgent` | `control_improver.py` | For each gap (especially policy/procedure mode), propose a new or enhanced control (15-field schema). |
| `source_type_prompts.py` | helper module, **not an agent** | Provides per-source-type prompt fragments injected into the four LLM agents above (Regulation vs Policy_Requirement vs Procedure_Step). |

### 3.2 Graphs (`src/regrisk/graphs/`)

- **`classify_graph.py` (Graph 1)**: `START → init → ingest → classify_group (loop) → end_classify → END`. Ingest dispatches on workbook shape: presence of a `Source_Inventory` sheet → `policy_parser.parse_policy_excel` (Policy/Procedure mode); otherwise `regulation_parser.parse_regulation_excel`. Both paths converge on the same `ClassifiedObligation` shape.
- **`assess_graph.py` (Graph 2)**: `START → map_group (loop) → prepare_assessment → assess_coverage (loop) → prepare_risks → extract_and_score (loop) → prepare_improvements → propose_improvement (loop) → finalize → END`. Conditional self-loops are implemented via `add_conditional_edges` with predicates like `has_more_map_groups`.
- **`graph_infra.py`**: shared infra — single `EventEmitter`, agent registry/cache, event loop reuse, tracing transport installer.
- **State**: `classify_state.py` / `assess_state.py` are LangGraph `TypedDict`s, not Pydantic. Pydantic models (`core/models.py`) are used for ingest, validation, and export.

### 3.3 Configuration

`core/config.py` loads `config/default.yaml` into a Pydantic
`PipelineConfig`. The YAML carries domain knowledge only (categories,
scoring scales, thresholds, UI tab visibility); LLM credentials come
from environment variables loaded via `python-dotenv` at startup
(`ui/app.py` calls `load_dotenv()`).

`config/risk_taxonomy.json` is loaded separately as a free-form dict and
threaded through agent contexts.

### 3.4 Tracing (`src/regrisk/tracing/`)

- `db.py` — `TraceDB` wraps a single SQLite file at
  `data/traces.db`. Tables include `events`, `llm_calls` (with
  `validation_passed`, `parsed_output`, `retry_attempt`), `run_metrics`,
  `run_comparisons`.
- `transport_wrapper.py` — `TracingTransportClient` wraps the LLM
  transport so every call is recorded with input/output, tokens, and
  cost estimate.
- `decorators.py` — `trace_node` decorator wraps every graph node when
  `trace_db` is passed to `build_classify_graph` / `build_assess_graph`.
- `listener.py` — `SQLiteTraceListener` listens to `EventEmitter` events
  and persists them.

Tracing is opt-in but turned on by the UI by default (`upload_tab.py`
attaches a `TraceDB` to every run).

### 3.5 Ingest (`src/regrisk/ingest/`)

- `regulation_parser.py` — reads `Requirements` sheet; groups by Citation Level 2 / 3.
- `policy_parser.py` — reads `Source_Inventory` sheet; normalizes `source_type` (Policy / Standard / Procedure); groups by `parent_source_id`. Exports `detect_source_inventory(path) -> bool` used by `classify_graph.ingest_node` and `upload_tab._detect_data_files`.
- `apqc_loader.py` — APQC PCF hierarchy parser + summary builder.
- `control_loader.py` — discovers and merges control workbooks.

### 3.6 Deterministic review (`src/regrisk/core/review.py`)

A dependency-free library exposing five rule sets:
`review_source`, `review_classification`, `review_mapping`,
`review_coverage`, `review_risk`, plus `review_procedure_contradictions`
and a `stamp_review(artifact, reasons)` writer that sets
`needs_review` / `needs_review_reasons`. The 14-reason catalogue is
imported as constants from `core/constants.py`.

**Status: library exists, not wired into any graph node.** Pydantic
models already carry `needs_review` / `needs_review_reasons` fields, and
the Streamlit "Needs Review Queue" UI is presumably designed to consume
them, but no `stamp_review` call is invoked during graph execution. See
gap #1.

### 3.7 Export (`src/regrisk/export/excel_export.py`)

Writes the multi-sheet output workbook consumed by reviewers. Sheet
inventory is per the flowchart and confirmed against the code. (Not
audited line-by-line for sheet counts — assume current.)

### 3.8 Streamlit UI (`src/regrisk/ui/`)

Tabs are registered in `app.py:_TAB_REGISTRY` and filtered by
`config.ui.visible_tabs`. Current registry (after the earlier
Executive View removal): Upload & Configure, Classification Review,
Mapping Review, Coverage, Risk Register, Traceability, Evaluation.
`config/default.yaml` still lists `"Executive View"`, which is now
silently dropped by the filter — should be reconciled.

`data_explorer_tab.py` exists in `src/regrisk/ui/` but is **not imported
by `app.py` and never registered as a tab.** Dead module — flagged for
owner decision (delete vs revive).

### 3.9 Demo loading contract

There is no auto-startup preload. The "demo" path the user must
preserve is the **`render_checkpoint_load(...)` dropdown** in
`upload_tab.py` (and other tabs via `components.render_checkpoint_load`)
which lists every JSON file in `data/checkpoints/`, lets the user pick
one, and calls `apply_checkpoint(data)` which writes every checkpoint
key into `st.session_state` and triggers `st.rerun()`. This bypasses
both graphs entirely.

There is also a private `_load_demo_data()` in `upload_tab.py` that
auto-picks the newest `STAGE_ASSESSED` checkpoint (preferring patched
ones) — but **it is never called from the UI** (its call site was
removed; only the helper remains). It is effectively dead code today.
The canonical demo checkpoint (per most-recent patched) is
`data/checkpoints/Improved_Patched_Full_Assessment_2026-04-13_22h56_43_2026-04-28_18h54_20.json`.

**No-touch zones (visible output):** the checkpoint loader, the JSON
schema of checkpoints, and the tab renderers that read from
`st.session_state` must produce byte-for-byte equivalent output after
the cleanup.

### Gap to fill in Phase 4

**GAP #1 — Deterministic review layer not wired into graphs.**

- **Feature:** Per-artifact `needs_review` flags and `needs_review_reasons`.
- **Current behavior without preload:** All artifacts come out of the live pipeline with `needs_review=False` and empty `needs_review_reasons`. The "Needs Review Queue" Excel sheet (and any UI section that filters on the flag) is empty for live runs.
- **Target behavior matching preloaded output:** Every classification, mapping, coverage assessment, scored risk, and proposed improvement is stamped by `core/review.stamp_review(...)` with reasons from `review_classification` / `review_mapping` / `review_coverage` / `review_risk`. Source-level reasons from `review_source` are propagated onto downstream artifacts derived from that source.
- **Proposed design (reuses existing patterns):** Add a tiny per-node post-step (not a new agent — review is deterministic, no LLM). Concretely: extend `classify_graph.classify_group_node` and the four `assess_graph` agent nodes (`map_group_node`, `assess_coverage_node`, `extract_and_score_node`, `propose_improvement_node`) to call the appropriate `review_*` function on each produced dict and `stamp_review` the result. No new graph nodes, no new state keys, no new infrastructure. One commit per node touched, or a single `feat: wire deterministic review layer into graphs` commit if changes stay small.

No other gaps were found severe enough to warrant Phase 4 work. Items
flagged for documentation (not code) in `docs/known-gaps.md` if any
emerge during Phase 4 implementation.

Proposed seed ADRs (specific to what was actually found, not generic):

- **0001 — LangGraph orchestration with two graphs bridged by `st.session_state`.** Why two graphs (human review checkpoint between them) instead of one.
- **0002 — Config-driven agents with deterministic fallback.** Why every agent works without an LLM (demo-grade, deterministic CI).
- **0003 — SQLite-backed tracing with per-node decorator + transport wrapper.** Why local SQLite instead of an external tracer; cost & quality fields.
- **0004 — Deterministic review layer as pure library.** Why human-review decisions never go through an LLM (auditability).
- **0005 — Checkpoint loading is the demo contract.** Why the dropdown bypasses graphs and what shape a checkpoint must have.
