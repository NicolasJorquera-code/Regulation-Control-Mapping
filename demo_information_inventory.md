# Risk Inventory Frontend Data Handoff

This is the short handoff for a risk-specialist ASI agent.

The goal is simple: generate better mock data for the Risk Inventory Builder frontend. The agent should understand what the workflow consumes, what the workflow produces, what the current mock-data quality is, and exactly what to return so Codex can update the YAML fixtures.

## 2. What The Workflow Consumes

The frontend workflow consumes four groups of data.

| 01 | Operating Context |
| --- | --- |
| Data points | Business units, process owners, products/services, systems, upstream/downstream dependencies, data objects, stakeholders, material obligations |
| Main files | `workspace_financial_institution_v2.yaml`, `packs/business_units_fi_v2.yaml`, `packs/processes_fi_v2.yaml`, `packs/regulatory_obligations.yaml` |
| Must be true | Business units and processes must use the active `BU-*` and `PR-*` IDs. |

| 02 | Process Evidence |
| --- | --- |
| Data points | Process narratives, policies, procedures, PDFs/Markdown/TXT summaries, control cues, risk cues, volumes, exceptions, reconciliations, approvals, testing results |
| Main files | Process fixture YAML files under `sample_data/risk_inventory_demo/` |
| Must be true | Each fixture must describe one concrete process and must not read like generic bank filler. |

| 03 | Control Baseline |
| --- | --- |
| Data points | Control ID, Control Level 1, Control Level 2, Full Description |
| Main files | `section_1__controls.xlsx`; process fixture `controls` sections for risk mapping detail |
| Must be true | The Knowledge Base Controls Register should not show business-unit fields or hierarchy nodes. Every risk-to-control mapping must still point to a real control in the same fixture. |

| 04 | Risk Framework |
| --- | --- |
| Data points | Risk taxonomy, root-cause taxonomy, control taxonomy, KRI library, risk appetite, scoring rules |
| Main files | `config/risk_inventory/*.yaml`, `packs/kri_library.yaml`, `packs/risk_appetite_framework.yaml` |
| Must be true | Use only supported `RIB-*` taxonomy IDs. User-facing prose should say `Frequency`, even though YAML uses `likelihood_score`. |

## 4. Current Mock Data Quality

Use the current data as a scaffold, not as the target quality bar.

| Area | Current state | Needed improvement |
| --- | --- | --- |
| Workspace | Good v2 operating model with 13 BUs and 60 processes. | Preserve it. Do not collapse the demo into a tiny sample. |
| Run fixtures | 10 hand-authored, 50 fallback-generated. | Prefer hand-authored fixtures for all 60 processes. If staged, improve the 10 priority fixtures first and clearly mark the rest as deferred. |
| Default process | `PR-16` is the best current example. | Match or exceed this quality for every authored fixture. |
| Sidecars | Evidence, issues, and obligations exist but do not load cleanly because of legacy IDs. | Rewrite sidecars with active `PR-*` process IDs and valid control/risk IDs. |
| Risk statements | Some are strong, especially payment exception handling. | Make every material risk concrete, process-specific, and executive-readable. |
| Controls | Many controls exist, but fallback runs can feel repetitive. | Controls should have clear owners, frequency, evidence, design rationale, operating rationale, and root-cause coverage. |
| KRIs | 51 load today. | Keep numeric thresholds, owners, data sources, escalation paths, and executive placement guidance. |

Best current sample to imitate: `RISK-PAYEXC-001` in `payment_exception_handling.yaml`.

Why it works:

- It describes a specific high-value payment release failure.
- It ties the risk to a real process chain: exception case, payment notice, generated wire file, approval, release, recall, reconciliation.
- It has root causes, impact rationale, frequency rationale, mapped controls, evidence, risk appetite, action plan, and review challenge.

What still needs improvement:

- The fixture still starts from legacy process ID `PROC-PAY-EXCEPTION` and is remapped to `PR-16` by the loader.
- Sidecar evidence, issues, and obligations need active `PR-*` joins.
- Similar quality should exist across every priority process, not only payment exception handling.

## 7. Data Each Process Fixture Must Include

Each process fixture should contain these sections.

| Section | What to include |
| --- | --- |
| `risk_appetite_framework` | Appetite statement, thresholds, escalation rules, approver, effective date |
| `scenario` | Active `PR-*` process ID, process name, business unit, product/service, systems, stakeholders, description |
| `controls` | 7 to 12 controls with owner, frequency, description, design/operating ratings, rationales, evidence quality, open issues, root-cause mapping |
| `risks` | 6 to 8 material risks for priority fixtures; each tied to taxonomy, controls, evidence, scoring, appetite, review, and action plan |
| `executive_summary` | Headline, key messages, top residual risks, recommended actions |

Each risk should include:

- risk ID
- `taxonomy_node_id`
- risk description
- risk event
- causes
- consequences
- affected stakeholders
- impact scores and rationales
- `likelihood_score` and frequency rationale
- mapped controls
- 4 to 6 exposure metrics
- 2 to 4 evidence references
- risk appetite status
- recommended action
- action plan
- one review/challenge record

## 9. Return Format

The ASI should return:

1. Scope: `full 60-process pack` or `staged 10-process pack`.
2. Counts: business units, processes, fixtures, risks, controls, KRIs, evidence artifacts, issues, obligations.
3. Business-unit risk distribution by L1 category: Operational, Cyber, Regulatory Compliance, Talent.
4. Fixture coverage table: process ID, process name, fixture path, risk count, control count, high/critical residual count.
5. Complete changed files using this block format:

````text
FILE: sample_data/risk_inventory_demo/<path>.yaml
```yaml
<complete file contents>
```
END_FILE
````

6. Validation results.
7. Assumptions and any deferred scope.

## 10. Validation Checklist

Before returning the data, the ASI must check:

- All YAML parses.
- Every in-scope process has exactly one run fixture.
- Every fixture `scenario.process_id` uses an active `PR-*` process ID.
- Every sidecar evidence, issue, and obligation record uses active `PR-*` process IDs.
- Every risk uses a valid `RIB-*` taxonomy ID.
- Every mapped control exists in the same fixture.
- Every evidence reference has a matching evidence artifact.
- Every High or Critical residual risk has an action, issue, control weakness, appetite breach, or clear rationale.
- Every risk has exactly one review/challenge record.
- No real customer data, real employee names, confidential exam findings, or real institution data.
- The existing loader can ingest the files without code changes.

That is the whole ask: make the mock data cleaner, more joined-up, and more executive-realistic, then return the exact files needed for Codex to update the repository.
